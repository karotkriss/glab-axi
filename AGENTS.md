# Project agent memory

This file is the project's committed home for project-intrinsic agent knowledge: build, test, release, architecture, and sharp-edge notes that should travel with the code.

## What this is

`glab-axi` is an AXI-compliant CLI that wraps the GitLab `glab` CLI for autonomous agents. It mirrors `gh-axi`'s architecture: token-efficient TOON output, minimal default schemas, contextual `help[]` suggestions, idempotent mutations, and structured errors on stdout. TypeScript/ESM, Node >= 20.

## Build / test / validate

- `npm run build` - `tsc` to `dist/`.
- `npm test` - vitest (unit tests mock `src/gl.ts`; no network).
- `npm run lint` - eslint. `npm run format:check` - prettier.
- `npm run skill:build` - regenerate `skills/glab-axi/SKILL.md` from the CLI's description + top-level help. `npm run skill:check` - fail if it has drifted (wired into CI; commit the regenerated file). The generator (`src/skill.ts`) pins every `npx -y glab-axi` invocation to the exact `package.json` version (`@X.Y.Z`), closing the skills.sh unbounded-download audit flag; this is why the release flow must `npm run skill:build` and commit `SKILL.md` alongside the version bump (see `glab-axi-release` skill), or `skill:check` fails CI.
- `npm run dev -- <args>` - run the CLI from source via tsx.

## Architecture

- `src/cli.ts` - the entrypoint. `main()` wraps `runAxiCli` in its own try/catch: `resolveContext` runs before the command handler and isn't covered by `runAxiCli`'s own try/catch, so an error thrown while resolving the target project (e.g. `glNotInstalledError` when `glab` is missing from PATH, surfaced through `isKnownGitLabHost`) is caught here and rendered through the same structured-error path instead of crashing as an unhandled rejection.
- `src/gl.ts` - the only place that shells out. Targets GitLab through `glab api <path>` (REST passthrough), never per-subcommand flags. `glApi` (parsed JSON), `glRaw` (raw text, e.g. job traces), `glApiResult` (no-throw, for idempotent delete). Host is targeted via the `GITLAB_HOST` env var; the project is addressed by its URL-encoded path inside the REST path (`projects/<encoded>/...`). One non-`glab` exception lives here: `runJq` pipes JSON through the system `jq` binary, backing `api --jq` (a missing binary returns `stderr === "ENOENT"`). A large `-f`/`-F` value (e.g. `repo create-file --content-file` on a big file) can blow the OS argv limit; `run()` maps the resulting spawn-time `E2BIG` to a `VALIDATION_ERROR` (`argumentTooLargeError()` in `src/errors.ts`) the same way it maps `ENOENT`, rather than surfacing an opaque `UNKNOWN`.
- `glApiList` (in `src/gl.ts`) is the list counterpart to `glApi`: it adds `-i` and returns `{ data, total }`, where `total` is GitLab's `X-Total` header - the only place the "how many are there" of AXI clause 4 is exposed. Feed it to `formatCountLine`'s `totalCount` to render `count: N of T total`. Two constraints: GitLab OMITS `X-Total` past 10,000 rows, so `total` is `null` there and the count line must fall back rather than invent a number (see the never-report-unverified-state rule below); and it is only for endpoints whose rendered rows ARE the server's result set, so `variable`/`secret` keep plain `glApi` because they partition on `masked` client-side and `X-Total` would count both surfaces.
- `api --jq <expr>` / `--raw` (alias `--json`) are opt-in escape hatches that operate on the RAW response (not the noise-stripped TOON view): `--jq` runs `jq -r`, `--raw` prints the JSON verbatim. Bare `api <path>` still emits stripped TOON. The `--jq`/`--json` plumbing is shared in `src/machine.ts` (`takeMachineFlags`, `applyJq`, `renderMachine`) and reused by `mr list`/`mr view` - both emit the raw GitLab response and bypass their schema flags (`--full`/`--comments`/etc.) when `--json`/`--jq` is set.
- **Never pass `-R` to `glab api`** - it rejects `-R`, and this previously broke `api` and `ci log` under `-R` targeting. The host goes through the env, the project through the path.
- `src/context.ts` - `resolveRepo(flagValue?, hostFlag?)`: priority for the PROJECT is `--repo` flag > git remote origin. The HOST layers on top: `--host` (explicit selector, wins) > `GITLAB_HOST` env > the host carried by `-R`/the remote. `-R` accepts `[host/]group/project` (nested group paths supported) and stays strictly project-addressing: a host-only `-R <host>` no longer silently falls through to the default host - `resolveRepo` THROWS `badRepoArgError` (VALIDATION_ERROR) pointing at `--host`, since a dropped flag returning wrong-instance data is the AXI clause-6 failure. `--host <host>` (a global flag stripped in `parseRepoContextArgs` alongside `-R`, so it never reaches `rejectUnknownFlags`) yields a host-only context: `RepoContext.project` is OPTIONAL, and `projectId`/`requireProject` guard on `ctx?.project` so host-level ops (`search`, `project list`, `api user`) target a self-hosted host without the `GITLAB_HOST` env workaround, while project-scoped commands still fail loud via `requireProject`. `parseRepoArg` is unchanged (still returns undefined for a single segment); the throw lives in `resolveRepo`. `GITLAB_HOST` still by itself does not select a project.
  `parseRepoArg` disambiguates host-from-group by segment count and known hosts, never by punctuation: two segments are ALWAYS `group/project` (a dotted first segment is a namespace, not a host - `firstname.lastname` is the standard username shape on LDAP/SSO instances), and only a 3+-segment value can lead with a host.
  `src/hosts.ts` (`knownHosts`) supplies the authoritative host set from `glab`'s own config plus `GITLAB_HOST`; it is what catches a bare intranet hostname with no dot to find.
  A dot is the last-resort tiebreak, kept so a host reachable via `GITLAB_TOKEN` but absent from that config stays addressable.
  Residual ambiguity, decided deliberately: a dotted TOP-LEVEL group that also has subgroups reads as a host, because `host/group/project` and `group/subgroup/project` are the same shape and nothing in the string can separate them. Personal namespaces cannot have subgroups, so the dotted-username case is unaffected.
- A git remote is only accepted when its host is one GitLab is known to be reachable on (`isKnownGitLabHost` in `src/context.ts`); an unknown host resolves to NO project rather than a guess. GitLab is self-hostable, so there is no hostname to allowlist - the signal is the wrapped CLI's own config, probed offline via `glConfigGet("api_host", host)` in `src/gl.ts` (`api_host` is set per host with no global fallback, so an unconfigured host reads back empty). `GITLAB_HOST` counts as explicit intent, which also covers a token-from-env setup with no config file (CI). Read only value-safe keys this way, never `token`. Without this, any forge's remote (github.com, bitbucket.org) parsed cleanly and was reported as a GitLab project that never existed. `parseRemoteUrl` stays a pure parser: validation belongs at the resolution boundary (`parseGitRemote`), not in the URL parser.
- `src/commands/*.ts` - one file per domain (issue, mr, ci, project, repo, label, variable, secret, release, search, api, auth, config, home, setup). `mr.ts` is the reference template.
- `variable` and `secret` are two views onto GitLab's single CI/CD Variables API (`projects/:id/variables`). `variable` = plain (unmasked) variables (list/get/set/delete, values shown); `secret` = masked+protected variables (list/set/delete, `list` never renders the value). The `list` commands partition by the `masked` flag so the two surfaces never overlap. `secret.ts` reuses the shared core exported by `variable.ts` (`variablesPath`, `upsertVariable`, `deleteVariable`, `requireName`) - intentional cross-command reuse. `set` is an upsert (GET-first: PUT if the key exists at that env scope, else POST; when the GET proves the value and flags already match it skips the write and reports `already: true`, so `set` never claims `updated` for an update that changed nothing - fixing it in `upsertVariable` fixes both surfaces at once). Keys are addressed per environment scope via `?filter[environment_scope]=<scope>` (default `*`).
  `variable set` takes its value from `--value` or piped stdin (`src/stdin.ts`); `secret set` is stdin-ONLY and refuses `--value` with a guiding error (`resolveSecretValue` in `secret.ts`), because argv is visible in process listings - do not "unify" it back onto the shared `resolveValue`.
  `--value` stays declared in `SECRET_HELP`'s `flags{set}` block so the refusal guides instead of collapsing into a generic unknown-flag error.
- `mr view`/`mr checks`/`mr diff` accept a full MR URL in place of the IID; the URL's own host/project target the request (precedence: `-R` flag > URL > git remote), tagged `source: "flag"` so it flows through to help suggestions like an explicit `-R`.
- `mr checks` reuses `ci.ts`'s exported `resolveMrPipeline`/`fetchJobs`/`renderSummary` rather than reimplementing verdict bucketing - intentional cross-command reuse.
- `mr merge` diagnoses a refusal instead of passing GitLab's opaque "Branch cannot be merged" through: `mergeBlocker()` in `src/commands/mr.ts` maps `detailed_merge_status` (with the coarse `has_conflicts`/`draft`/`blocking_discussions_resolved` flags as fallback for instances that omit it) to a cause plus the command that clears it. It costs no extra call - it reads the MR the idempotency GET already fetched, or, when `--rebase` ran first, the fresher post-rebase snapshot `waitForRebase` returns (the pre-rebase GET would diagnose a state the merge already moved past). A cause it cannot name keeps GitLab's own message rather than inventing one, but still carries a help[] line: an error with no resolving suggestion IS the AXI clause 9 failure.
  A blocker also carries the error `code` its cause proves (conflict -> `CONFLICT`, matching what `waitForRebase` already throws, so both paths of `mr merge` report one conflict identically).
  That code is set HERE, at the verb layer, and deliberately NOT by a `405` pattern in `mapGlError`: GitLab refuses a merge with **HTTP 405**, whose body carries neither a cause nor the request path, and the byte-identical 405 also answers a merge of an ALREADY-MERGED MR (verified live, 2026-07-15).
  So a blanket `405 -> CONFLICT` in the pattern table would confidently mislabel a non-conflict - reachable today via `api PUT .../merge` - and no pattern can scope itself to the merge endpoint, because `mapGlError` never sees the path.
  Only this function knows both that it is a merge and what the server actually said the state was, so the code comes from the MR payload we verified, never from the status. Do not "simplify" it into `errors.ts`.
  A blocker with no proven code keeps the mapped code rather than guessing: `CONFLICT` is claimed only where the state proves a conflict, not for `not_approved`/`ci_must_pass`/draft.
- `ci list` orders by `id` (GitLab's own default for the endpoint), NOT `updated_at` like the other list commands, and that inconsistency is deliberate - do not "restore" it.
  `order_by=updated_at` makes GitLab answer **500** on a project with a huge pipelines table, so it broke on exactly the projects with the most CI history (verified live against `gitlab-org/gitlab`, 2026-07-15; `sort=desc` does not help, and the same query is fine on a small project).
  The scaling limit is pipeline-specific, not a shared pattern: `issue list`, `mr list` and `home` issue the same `order_by=updated_at` and were each driven against that project unfiltered (`--state all`) without a 500, so they keep their ordering - a blanket removal would change their behaviour to fix a bug they do not have.
  The cost of `id` over `updated_at` is real but small: a pipeline retried long after creation sorts by when it was created, not when it last changed.
- The `stuck` section (`renderStuck` in `src/commands/ci.ts`, emitted from the shared `renderSummary`, so `ci status`/`ci view`/`ci jobs`/`mr checks` all get it) names a pending job no active runner can pick up - blocked, not queued. REST carries NO `stuck` flag on a job (only GraphQL's `CiJob.stuck`, which has no reason), so it is derived from the job's `tag_list` plus `GET projects/:id/runners?status=online&paused=false&tag_list=<tags>`; that filter is AND semantics, matching GitLab's own job-to-runner rule, so the verdict is the server's. The lookup only fires when a job is `pending`, and once per distinct tag set. `verdict` deliberately keeps its three values (a stuck pipeline has not finished, and pollers branch on that field) - the signal is additive, read from the section's presence. An unreadable runner list renders `stuck: unavailable - <reason>`, never a silent "not stuck". Known gap, marked `ponytail:` in the source: an UNTAGGED job also needs a runner accepting untagged work, and `run_untagged` is detail-only on the runners API, so an all-tag-only instance under-reports (a miss, never a false positive).
- `ci log` strips ANSI at the output boundary (`stripAnsi` in `src/commands/ci.ts`) because GitLab traces are dense with colour/section escapes that cost tokens and carry no meaning; strip BEFORE measuring, so `original_length` and the tail slice describe the log the agent is actually shown. A truncated trace also spills to a temp file (`spillLog`) reported as `full_log`, so `--full` is not the only escape - the agent greps the file instead of paying for 350KB of context. The path is rendered only when the write succeeded; a failed spill must not fail a read-only command. The spill lives in one stable per-user dir (`SPILL_DIR`, keyed by job id) so a repeat `ci log` on the same job overwrites its file instead of stranding a fresh dir every call; `ensureSpillDir()` guards it before writing - `lstatSync` never follows a symlink, and the dir must be owned by the current uid with no group/other access (`mode & 0o077 === 0`) - falling back to a fresh `mkdtemp` dir, never through an unsafe stable path, since a trace can carry leaked CI secrets.
- `mr diff` uses the MR `/changes` endpoint (single call; its `overflow` flag marks a server-truncated diff on huge MRs), not the paginated `/diffs`. Default is a bounded per-file summary (path, status, `+`/`-` counts derived by counting hunk lines); `--full` reconstructs the `diff --git`/`---`/`+++` headers GitLab omits, since each `changes[].diff` is only the hunk body (starts at `@@`).
- `mr view --reviews` folds `/approvals` + `/discussions` into a review summary: it derives the `approved` bool when GitLab CE omits it (`given >= approvals_required`), and counts only *resolvable* threads (plain comments are excluded from the resolved/unresolved totals).
- `mr merge --auto` (gh-axi `pr merge --auto` parity) sets `merge_when_pipeline_succeeds=true`; GitLab merges immediately when there is no pipeline / it already passed (state `merged`, rendered as `merged`) else defers (state stays `opened`, rendered as `auto_merge: enabled`). It refuses to combine with `--rebase` (auto-merge can't rebase) and skips the eager rebase-poll path.
- `ci watch <pipeline-id>` polls until the pipeline is terminal, then prints the same verdict aggregate as `ci status`. Terminal detection lists the ACTIVE statuses and treats everything else as terminal, so an unknown GitLab status can't spin the loop forever; polling is bounded by poll count (`timeout / interval`), not wall-clock.
- `ci watch`'s exit code follows the pipeline's own status, not the job verdict (a canceled pipeline with passing jobs still exits non-zero): success -> 0, anything else -> `process.exitCode = 1`, since throwing an `AxiError` would replace the verdict output on stdout. The delay goes through `src/sleep.ts` so tests can mock it away instead of waiting real seconds.
- `ci run` is pipeline dispatch (gh-axi's `workflow run`): `POST projects/:id/pipeline` with `ref` defaulting to the project's default branch via `resolveDefaultBranch` (exported from `repo.ts`). `--field KEY=value` emits key-then-value per variable, the same Rails param-grouping rule as release assets. GitLab has no workflow entity - the ref decides what runs - so there is no `workflow` noun to list/enable/disable.
- `ci cancel` GET-firsts and skips the POST when the pipeline is already terminal, reusing `ci watch`'s `isPipelineTerminal` rather than depending on how a given GitLab version answers a cancel on a finished pipeline. It reports the status it read (`success` stays `success`), never a hardcoded `canceled`.
- `label edit`/`release edit` are PUT-only, deliberately skipping the GET-first idempotency convention: the PUT response is itself the verified state, and for `label edit --name` (a rename) a GET-first would find the old name gone on a repeat and report `already_absent`, where the honest answer is GitLab's own "label not found".
- `release create` mirrors gh-axi's flags onto the GitLab Releases API, which lacks direct equivalents for some (see `release --help` notes + `src/commands/release.ts` for the authoritative mapping): `--target` aliases `--ref` -> `ref`; `--prerelease` -> a far-future `released_at` (GitLab renders it "upcoming"); `--asset <url>[#name]` -> `assets[links][]` (links to hosted URLs, not file uploads; emit name-then-url per asset so Rails groups the pairs). `--draft`/`--generate-notes` have no GitLab concept and refuse with a `VALIDATION_ERROR` (exit 2) rather than silently no-op - a decided product call, not a stub.
- `project create [namespace/]name` is the one project subcommand that does NOT `requireProject` - it creates a new project via `POST /projects`, using `ctx` only for host targeting. An org-owned create resolves the leading namespace to `namespace_id` via `GET /namespaces/<path>` (the create API takes the integer id, never a path). Idempotency is GET-first (not catch-the-conflict like label/release): GitLab's "has already been taken" is an object-valued 400 whose message the `glab` unmarshal quirk masks to a generic string (see `src/errors.ts`), so the message-regex approach can't detect it - a GET on the target path can. `--clone`/`--template` are gh concepts with no clean `glab api` path and refuse with `VALIDATION_ERROR` guidance; visibility defaults to `private`.
- `project` vs `repo`: `project` addresses the GitLab project entity (`/projects/:id`), `repo` writes its git contents (`/projects/:id/repository/*` - `create-file`, `create-branch`). The split mirrors GitLab's own API namespaces and keeps the two-level `<domain> <subcommand>` shape; new repository-write surfaces (commits, tags) belong in `repo.ts`. `repo`'s `--branch`/`--ref` default to the project's default branch via one `GET /projects/:id` (`resolveDefaultBranch`), which errors actionably on an empty repository - the one case a branch must be named.
- `project delete` takes its target as a required positional (numeric id or `[host/]group/project`) and deliberately does NOT fall back to the resolved `ctx` project, so it cannot delete the project the agent happens to be standing in. It is the only destructive-confirmation command: `--yes`/`-y` is required and it never prompts (AXI forbids interactive prompts), so the pattern to copy for any future destructive command lives in `src/commands/project.ts`.
- `project delete` reports the outcome it read back, never a hardcoded one. GitLab answers `202 Accepted` whichever it does, so the DELETE is followed by a GET: where the instance enables delayed project deletion the project SURVIVES, renamed to `<path>-deletion_scheduled-<id>` with `marked_for_deletion_on` set, which renders `status: scheduled` + `purge_after` (plain `status: ok` is reserved for a real purge). That rename is why the read-back goes by NUMERIC id captured from the GET-first - the path just deleted now 404s while the project still exists. A project already marked is a no-op (`already: true`); deleting it again is a 400.
- `auth` (`src/commands/auth.ts`) is host-scoped, never project-scoped, and delegates to `glab auth git-credential` (the wrapped CLI's own credential helper) via `glCredential` in `src/gl.ts` - the second non-`glab api` shell-out after `glConfigGet`. It exists so nothing has to parse the CLI's config file for a clone credential; do not reimplement credential handling here, and do not add write/rotate/cache verbs (`store`/`erase` are passed straight through).
  `auth git-credential` is the ONE command that deliberately breaks two house rules, and both breaks are load-bearing: it emits git's raw `key=value` protocol instead of TOON, and it stays SILENT on failure (exit 1, empty stdout). Its consumer is git, which parses stdout as credential fields and rejects anything else as malformed - a structured error there would break the operation the verb exists to serve. The agent-facing diagnosis lives in `auth status` instead. Do not "fix" it to render TOON or a structured error.
  `auth status` never prints the credential; it spends it on a `GET /user` and reports the `account` the server named, since presence on disk does not prove a token still works (a failure renders `unavailable - <reason>`, per the never-report-unverified-state rule). `token` is presence-only (`present`/`absent`) - never a prefix, suffix, or length, because this output gets pasted into logs and status lines.
  `--host` scopes the report to one host; WITHOUT it, every configured host is enumerated rather than a default being guessed. No hostname is hardcoded: enumerating cannot answer confidently about the wrong host, which is the failure a single-host default would cause.
  `auth status` also reports the INSTALL (`glInstalls` in `src/gl.ts`): `bin`, `version`, `config_file`, `default_host`, plus a `shadowed` section listing every other same-named executable on PATH. This is the one place the wrapped binary's NAME is deliberately emitted (as a path), breaking the scrubTool house rule, and it is load-bearing: a tool that shells out owes its caller the identity of what it shelled out to. It comes from a real incident - two installs ~70 releases apart with separate config files and different default hosts, where every call silently drove the older one and diagnosing it required the bare CLI at every step. `test/auth.test.ts` asserts the no-leak rule against the prose only, excluding the `bin`/`path`/`config_file` fields.
  Installs are de-duplicated by REALPATH, or `/usr/bin` + `/bin` on a merged-usr system would report a shadowing conflict on nearly every Linux box. `shadowed` renders only when there is a genuine conflict, so its presence is the finding.
  `glConfigGet("host")` (no host argument) reads the GLOBAL default host - the one a call omitting `--host` lands on. That is the single most load-bearing setting here and nothing else reports it.
  Two facts worth knowing before touching it: `GITLAB_TOKEN`, when set, answers for EVERY host and overrides the per-host store, so host-scoping is not absolute; and help text here must not name the wrapped CLI's binary (only comments in `src/` do) - the non-interactive redirect is `GITLAB_TOKEN`, since login is interactive and AXI forbids prompts.
  Validating it needs an authenticated git server, not just an exit code: a verb that returns an empty credential exits cleanly. `test/auth.test.ts` covers the units; the end-to-end proof is a local `git http-backend` behind Basic auth, cloned through the helper (a real self-hosted instance is not required, and a live one should not be used as a test target).
- `config` (`src/commands/config.ts`) is read-only introspection of the wrapped CLI's own configuration, host-scoped via `--host` (omit it for the global scope), and reports WHICH scope answered - global and per-host settings disagree in ways that decide where calls land.
  Any key whose name contains `token` is refused BEFORE the read, so the value never enters the process. This is not theoretical: `glab config get -h <host> token` returns the live token on stdout (verified 2026-07-20). The substring match is deliberate over an exact one - it has to hold for whatever key the wrapped CLI adds next, because the cost of being wrong is an unrecoverable credential leak. Presence is the whole question, and `auth status` answers it.
  It reads through `glConfigGetResult` (`src/gl.ts`), the no-throw sibling of `glConfigGet`: the resolution path is happy collapsing "unset" and "read failed" into `""`, but a command that REPORTS the value to an agent must not, since rendering a failed read as `unset` is exactly the confident lie the never-report-unverified-state rule bans.
  `scope` renders as `host <host>`, not `host:<host>` - a colon makes TOON quote the whole value.
- `src/refusals.ts` - the deliberate-refusal model, applied at the VERB layer; `src/commands/release.ts` and `project.ts` hold the same model at the FLAG layer (`--draft`, `--template`).
  The point is disambiguation, not politeness: without it every unbuilt verb collapses to one "Unknown <domain> subcommand" string, and an agent cannot tell "GitLab has no such concept" from "not built yet" - opposite facts.
  Each entry gives a reason plus runnable redirects, in three kinds: divergence (no GitLab concept -> point at the real model), renamed (`mr close` -> `mr update --close`), unbuilt (real endpoint, unwritten -> hand over the `api` call; `{project}` is expanded by `api`).
  **When you build a refused verb, delete its entry** - `test/refusals.test.ts` fails if a refusal survives its own implementation, and it also asserts every suggestion names a real verb and never redirects to another refused one. That test is the guard against shipping guidance the CLI's own parser rejects; harvest and run what you emit before trusting it.
  Fork routing is a deliberate out-of-scope gap (captain, 2026-06-25: GitLab scope is direct branch->MR only) and is intentionally absent from the table.
  `refuseSubcommand()`'s third argument is a `string | { message, help }` union: pass the domain's own `<DOMAIN>_HELP` text (the common case, one token at each of the 10 router call sites) to have the generic "unknown subcommand" error inline the valid subcommands parsed from that help; pass the old `{ message, help }` shape instead when a fallback needs custom wording (`search` keeps its own "Unknown search type" fallback since its list is "types", not "subcommands"). Either way the fallback still only fires when `REFUSALS[domain][sub]` has no deliberate entry.
- IID-addressed: issues and merge requests use their project-scoped IID (the number in the URL), not the global id.
- `glab` field flags are inverted from `gh`: `-F`/`--field` = typed, `-f`/`--raw-field` = raw string. In `GlApiOptions`, `fields` -> `-F` (ids/booleans), `rawFields` -> `-f` (user text: titles, bodies, labels).

## Conventions

- ESM NodeNext: all local imports use the `.js` extension.
- Validation errors `throw new AxiError(...)` (the harness renders them). That includes an unknown subcommand: routers call `refuseSubcommand()`, which always throws, because a returned `renderError(...)` string exits 0 and an agent checking the exit code reads a mistyped verb as success.
- Numeric flags go through `parseLimit()` (raw `parseInt` yields `NaN` -> `per_page=NaN`).
- **A command's `<DOMAIN>_HELP` text is the runtime source of truth for which flags it accepts, not just documentation.**
  `rejectUnknownFlags()` (`src/args.ts`), called once for every command from `withRepoContext` in `src/cli.ts`, parses the `flags{sub}:` blocks and rejects anything a subcommand does not declare (AXI clause 6: a dropped flag is worse than an error, because the agent proceeds confidently on wrong data).
  So **adding a `takeFlag(args, "--new")` without adding `--new` to that subcommand's `flags{...}` block ships a flag the CLI rejects.** The upside is that this drift now fails loudly instead of silently, which is why there is no second flag registry to keep in sync.
  Three consequences worth knowing before editing a help table: prose inside a `flags{...}` block is parsed, so naming a flag there to say it does NOT apply makes it accepted (say that in `notes:` instead); a subcommand appearing in several blocks gets the union, never the last one; and a subcommand ALIAS must be listed in a flags key (`flags{edit,update}`) or its invocation goes unvalidated.
  A deliberately refused flag (`--draft`, `--template`) must stay declared, or its guiding refusal collapses into a generic unknown-flag error.
  `api` opts out - it is the raw passthrough and forwards arbitrary flags by design.
  Only a `--`-leading token is inspected as a flag candidate (matching `getPositional` and `search`'s own query-builder convention elsewhere in the codebase); a single-dash token is left alone as free text/positional data, since search queries and other positionals routinely start with a negative number, a version, or a word that happens to start with a dash. The accepted cost: an unknown single-dash flag goes unvalidated - there are almost none (`-R` is stripped earlier, `-y`/`-h` are known).
  A boolean flag (one read with `takeBoolFlag`, not `takeFlag`) must be marked in its `flags{...}` block with a trailing `!` right after the token, e.g. `--full!`. This drives a second check: the equals form (`--full=true`) is rejected as a `VALIDATION_ERROR` naming the flag, rather than being silently accepted by the guard while `takeBoolFlag`'s exact-match `indexOf` ignores it and the filter it was meant to set never applies. Forgetting the `!` on a boolean flag reopens that exact hole, so mark it whenever a new boolean flag is added.
- Mutations are idempotent: GET current state first; if already in the target state, return a no-op (exit 0). Delete-on-404 is a no-op.
- Structured errors must never leak the wrapped CLI's name - `scrubTool()` in `src/errors.ts` strips it; "GitLab" (the product) is fine.
- **The `glab` config file holds the API token, so nothing read from it may reach stderr.** `src/hosts.ts` is the only place that parses it, with yaml's `logLevel: "error"`: a yaml WARNING quotes the source line it is unhappy about, which is the token's own line, and a warning is not a throw, so the `try`/`catch` around the parse cannot stop it (this printed a live token to stderr on nearly every invocation, fixed 2026-07-15). Silence the channel, never the one construct that triggered it. Not `logLevel: "silent"` - that also silences parse ERRORS, so malformed yaml stops throwing and a half-parsed object becomes the host list, inventing hosts where the contract says report none. `test/hosts.test.ts` guards both halves.
- **Never report state that was not verified against the server.** Report what a response said, not what the request asked for, and never let a failure render as a value: a swallowed error that becomes `0` or a hardcoded `status: ok` is a confident lie, and an agent trusting this tool cannot tell it from the truth. A count is printed only when one was received (`renderSection` in `src/commands/home.ts` renders `unavailable - <reason>` otherwise); an outcome is read back before it is named (`project delete`). If the state is genuinely unknown, say so - "unavailable" and "0" are opposite facts. This is the one bar a 2026-07-15 live audit found the tool failing in two unrelated commands at once, so weigh it before adding any summary line.

## Releasing

- Every change adds an entry under `## [Unreleased]` in `CHANGELOG.md` (Keep a Changelog format). That section is the single source of truth for a version's GitHub release notes.
- Publishing is automated: `.github/workflows/release.yml` builds, runs the test suite, then runs `npm publish` on `release: [published]` only (not on push/tag).
- Auth is npm Trusted Publishing (OIDC) - there is no npm token in this repo, and none should be added. The registry mints a short-lived credential from the workflow's GitHub Actions OIDC identity, so the job needs `permissions: id-token: write`. The trusted publisher is registered on npmjs.com against this repo and the `release.yml` filename, so renaming that file breaks publishing until the registration is updated. `environment: npm-publish` stays on the job: it is part of the registration and allows environment protection rules on publishing.
- Trusted publishing requires npm >= 11.5.1 and Node >= 22.14.0. No Node 22 release bundles an npm that new (22.x ships npm 10.x), so the workflow pins Node 24, whose bundled npm clears the floor without an upgrade step. A "Verify toolchain" step asserts both floors against the resolved binaries, so the run fails loudly rather than reaching a doomed publish. `--provenance` is intentionally absent: trusted publishing signs provenance by default from GitHub Actions. These floors bind the CI publisher only - they are unrelated to the `engines` range the package supports for its users, so do not "sync" the two.
- `actions/setup-node` must stay at v7 or newer in `release.yml`: every earlier major exports a dummy `NODE_AUTH_TOKEN` (`XXXXX-XXXXX-XXXXX-XXXXX`) whenever `registry-url` is set, which muddies OIDC auth. v7 was the first release to drop it. Do not downgrade it, and do not add caching to that job - it is the repo's most privileged job.
- The same workflow has a `workflow_dispatch` trigger with a `dry_run` input (default true) that installs, builds, tests, and packs plus validates the tarball. It does NOT verify auth and must never be described as doing so: `npm publish --dry-run` never contacts the registry for credentials, so only a real publish exercises the OIDC handshake. (The pre-OIDC dry run asserted `npm whoami` proved the publish path worked; it passed green while the real publish died on `EOTP`, which is why that claim is gone.)
- The end-to-end cut-a-release procedure (CHANGELOG move, version bump file list, tag, `gh release create` with notes extracted from the CHANGELOG section, verify via `npm view glab-axi version`) lives in the `glab-axi-release` skill (`skills/glab-axi-release/SKILL.md`) - follow it rather than reinventing the steps.
- **A user's global install shadows any UNPINNED `npx -y glab-axi` forever** - the README's "Zero setup" flow and the bare `glab-axi` command (`setup hooks`) are still exposed, which is why README's "Global npm install" section carries a check-and-upgrade caveat. `npx` short-circuits: when `<globalBin>/glab-axi` exists it runs that binary and returns WITHOUT asking the registry for a version (`libnpmexec/lib/index.js`, the `fileExists(globalBin/...)` branch). So a stale global answers with old behaviour at exit 0 and nothing announces it - a released fix never reaches that user. The skill's own invocations are no longer exposed to this: `src/skill.ts` pins them to the exact `package.json` version (`npx -y glab-axi@X.Y.Z`), and a version-pinned request only reuses the global copy when its version matches the pin exactly (harmless - it's already current); otherwise `npx` fetches and runs the pinned version fresh, bypassing a stale global (verified against an isolated global prefix, 2026-07-17). Do not delete the README caveat as redundant - the unpinned surfaces remain exposed.
  Verifying this needs a real global prefix: set `prefix=` in an isolated `$HOME/.npmrc`, NOT `npm_config_prefix` in the env - npm treats `prefix` as protected when it comes from the env, `globalBin` collapses to `/bin`, and the shadow branch silently never fires, so `npx` fetches latest and the shadow looks fixed when it is not.

## Targeting a self-hosted instance

The tool is fully generic. Point it at any self-hosted GitLab with `GITLAB_HOST=<host>` (or `-R <host>/group/project`, or a git remote on that host). Do not hardcode any host anywhere.

Driving verification against a real instance is fine; naming it in anything public is not.
A real host, namespace, or project path must never land in tests or fixtures, examples, help text, commit messages, or PR bodies - use the placeholders this repo already establishes: `gitlab.example.com`, `dev.example.gy` (see `test/context.test.ts`), `group/project`, `my-group`.
This is cheap to honour up front and effectively irreversible once missed: a merged commit keeps the string forever, and editing a PR body leaves the prior revision visible in GitHub's edit history.

Removing such a string is the one case where the cleanup itself re-publishes it, so check the result rather than assuming.
A diff that deletes a hostname still contains it on every `-` line, and the gate writes the PR body from that diff: a remediation PR will happily narrate the exact string it removed. Grep the PR body after the run, not just the tree.
Because a real value is by then indistinguishable from a placeholder in the write-up, keep the two apart from the start - that is what makes this rule cheaper to follow than to unwind.
